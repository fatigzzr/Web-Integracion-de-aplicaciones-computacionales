Ejercicio 16: Prueba spike

Incremento abrupto de usuarios (ej. de 200 a 1000 en 10 segundos).

Introducción

En este ejercicio se evaluará el desempeño de los microservicios que conforman la aplicación PredictHealth, una plataforma orientada al análisis y predicción de riesgos en salud. El sistema está compuesto por diversos microoservicios que operan de forma coordinada, entre ellos:

  • auth: encargado de la autenticación y emisión de tokens.
  • register: gestión del registro de nuevos usuarios.
  • patient: administración de perfiles y expedientes de pacientes.
  • health: operaciones generales de información médica.
  • diabetes: motor de predicción y análisis de riesgo para diabetes.
  • hypertension: procesamiento de predicciones y evaluaciones de hipertensión.
  • data: manejo de datos clínicos.
  • recommendations: generación de recomendaciones personalizadas para el usuario.

Dado que PredictHealth depende de la interacción simultánea entre múltiples microservicios, resulta fundamental validar que cada uno mantenga un comportamiento estable, eficiente y escalable bajo distintos niveles de carga. Las pruebas de rendimiento permiten identificar cuellos de botella, medir tiempos de respuesta, determinar la tolerancia al tráfico inesperado y garantizar que el sistema pueda sostener un uso real en producción.

Estas pruebas son especialmente importantes en aplicaciones del sector salud, donde los fallos pueden impactar directamente en la disponibilidad y confiabilidad de la información médica. Por ello, la ejecución sistemática de pruebas baseline, smoke, ramp-up, spike, soak y break-point constituye un paso crítico para asegurar que los microservicios de PredictHealth operen correctamente ante escenarios exigentes y condiciones cambiantes de carga.

Objetivos Específicos

A continuación, se presentan los objetivos específicos de cada tipo de prueba de rendimiento que se aplicará a los microservicios del sistema:

5. Prueba Spike (incremento abrupto, e.g., 200 a 1000 usuarios en 10 s)

  • Medir la resiliencia del sistema ante picos súbitos de tráfico.
  • Identificar si alguna parte del sistema colapsa o rechaza solicitudes durante el pico.
  • Analizar si la infraestructura puede absorber aumentos repentinos sin afectar al usuario final.

Metodología

  1. Arquitectura del entorno de prueba: se configuró una arquitectura basada en microservicios independientes, cada uno ejecutándose en un puerto diferente en localhost. El generador de carga (Locust) interactuó directamente con los servicios mediante solicitudes HTTP.
  2. Herramientas utilizadas:
    • Locust: herramienta principal para la generación de carga y ejecución de pruebas de rendimiento.
    • Lenguaje de scripting: Python.
    • Sistema operativo: macOS, Linux o Windows.
    • Formato de resultados: archivos CSV generados y reportes HTML para visualización ejecutiva.
  3. Descripción de los escenarios de carga:
    • Prueba Spike
      • Usuarios: un incremento abrupto aplicado generalmente entre 50 y 200 usuarios
      • Tasa de spawn: muy elevada (entre 20 y 50 usuarios/segundo)
      • Descripción: Se aplicó un aumento repentino de usuarios que permitió evaluar la capacidad de los microservicios para soportar picos rápidos en el tráfico.

def get_service_url(service_name: str) -> str:
    """Build service URL from base host and port"""

class PredictHealthUser(HttpUser):
    """Base user class with authentication and common tasks"""
    
    abstract = True  # Don't run this directly
    host = "http://localhost"  # Base host (services use absolute URLs with specific ports)
    access_token: Optional[str] = None
    user_id: Optional[int] = None
    user_email: Optional[str] = None
    
    def on_start(self):
        """Called when a simulated user starts - perform login"""
        self.login()
    
    def login(self):
        """Authenticate and get access token"""
        # Pick random test user
        user_creds = random.choice(TEST_USERS)
        
        response = self.client.post(
            f"{get_service_url('auth')}/auth/login",
            json={
                "username": user_creds["email"],
                "password": user_creds["password"]
            },
            name="/auth/login"
        )
        
        if response.status_code == 200:
            data = response.json()
            self.access_token = data.get("access_token")
            self.user_email = user_creds["email"]
            
            # Get user ID from /auth/me
            if self.access_token:
                me_response = self.client.get(
                    f"{get_service_url('auth')}/auth/me",
                    headers={"Authorization": f"Bearer {self.access_token}"},
                    name="/auth/me"
                )
                if me_response.status_code == 200:
                    me_data = me_response.json()
                    self.user_id = int(me_data.get("sub", 1))
        else:
            # If login fails, use fallback
            self.user_id = 1
            self.access_token = None
    
    def get_auth_headers(self):
        """Return Authorization header dict"""
        if self.access_token:
            return {"Authorization": f"Bearer {self.access_token}"}
        return {}

        for csv_path in possible_paths:
            try:
                with open(csv_path, 'r') as f:
                    reader = csv.DictReader(f)
                    data = list(reader)
                    print(f"Loaded {len(data)} users from {csv_path}")
                    return data

                continue
        
        print("Warning: users.csv not found in any location, using default test users")

            if response.status_code == 200:
                data = response.json()

                if self.access_token:

                        f"{get_service_url('auth')}/auth/me",

Interpretación de resultados

  1. ¿Qué sistema es más sensible a picos de carga?

    El sistema más sensible al spike es auth, especialmente el endpoint /auth/login.

    Esto se confirma por:

    • Los tiempos de respuesta del percentil 95 en la gráfica suben de 5,000 ms a casi 40,000 ms, lo cual es una señal clara de saturación del servicio.
    • En el archivo stats_stats_history.csv, los primeros registros en mostrar latencias anormalmente altas (29,000–39,000 ms) coinciden con el momento en que sube la carga simultánea a 100 usuarios.

    Además, la autenticación es el único servicio cuyo tiempo no se estabiliza nuevamente después del spike.

  2. ¿Cuál es el punto de ruptura de cada servicio?

    El spike permite observar cuándo cada servicio deja de responder aceptablemente, incluso sin que existan failures explícitos.

    Los datos muestran lo siguiente: (de débil a más fuerte)

    • Auth/login: rompe entre 80 y 100 usuarios, con latencias >30–40 segundos (gráfica: pico morado del percentil 95).
    • POST de paciente y estilo de vida: tienen picos entre 7,000 y 15,000 ms, por lo que su ruptura ocurre después que auth pero antes de los GET.
    • GET (predicciones, paciente, recomendaciones): mantienen tiempos bajos (<3000 ms) incluso durante el spike; no muestran ruptura real en esta prueba.
  3. ¿El CMS se comporta igual que los microservicios bajo carga de escritura?

    Sí, el patrón observado confirma que las operaciones POST son las más afectadas por picos, mientras que los GET se mantienen estables.

Conclusiones y recomendaciones

Los resultados de la prueba Spike muestran que el sistema experimenta una degradación severa cuando la carga aumenta de manera abrupta. /auth/login no logra estabilizarse después del incremento repentino de concurrencia, lo que indica que su capacidad actual es insuficiente para absorber picos abruptos de usuarios concurrentes. En contraste, los endpoints de lectura mantienen un rendimiento relativamente estable, lo cual indica que el problema está más relacionado con operaciones de escritura y validación interna.

Aunque no se registran fallas explícitas, los tiempos de respuesta extremadamente altos implican que el sistema está muy cerca de producir timeouts y errores si la carga continuara aumentando. Esto define claramente el punto de ruptura, el sistema empieza a volverse inutilizable alrededor de los 80–100 usuarios concurrentes, donde auth comienza a responder con latencias superiores a 30–40 segundos.

Como recomendaciones, resulta esencial optimizar el servicio de autenticación mediante mejoras en el hashing, el manejo del pool de conexiones o la introducción de mecanismos de caching. La prueba Spike demuestra que el sistema puede manejar cargas graduales, pero no está preparado para picos.
