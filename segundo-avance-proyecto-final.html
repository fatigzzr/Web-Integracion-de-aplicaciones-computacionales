Módulo 1: Backend

Objetivo del Módulo 1 (Back End CMS)

El objetivo del módulo 1 es desarrollar un backend completamente funcional para administrar contenido, usuarios y datos del sistema de forma centralizada.

Funcionalidades implementadas

El backend de nuestro proyecto está desarrollado con Flask de Python, utilizando PostgreSQL para la base de datos, con autenticación mediante JWT y la configuración de CORS.

  1. Login con sistema de Autenticación y Autorización

    Utilizamos el endpoint /api/login para que los usuarios puedan iniciar sesión. El login cuenta con autenticación del usuario, si es administrador o no, verificando las credenciales con la base de datos. Para tener la seguridad implementada, realizamos un hash de contraseñas, un registro de IP de acceso, los Tokens de JWT con expiración configurada.

  2. Gestión de Usuarios

    Utilizamos el endpoint /api/me para obtener la información que está guardada de un usuario autenticado. Se accede a su ID, email, y rol. El endpoint implementa autenticación JWT validando el token Bearer y consulta la base de datos usando sp_get_user_data() para obtener datos actualizados del usuario. Es un checkpoint de autenticación que se ejecuta cada vez que el usuario abre la aplicación para asegurarse que sigue estando autenticado.

  3. CRUD

    Utilizamos diferentes endpoints para permitir a los administradores y analistas interactuar directamente con las tablas de la base de datos a través de la interfaz web. Utilizamos el api/entidades para listar todas las tablas dentro de nuestra base de datos, y en caso de querer obtener los datos dentro de la tabla, utilizando el método GET se utiliza el endpoint api/entidades/<nombre_entidad>. Si se utiliza endpoint api/entidades/<nombre_entidad> con el método POST, podemos insertar registros a la tabla deseada, y para actualizar los registros dentro de la tabla utilizamos el método PUT/PATCH con el endpoint /api/entidades/<entidad_name>/<pk_value>. Por último para eliminar registros usamos el endpoint /api/entidades/<entidad_name>/<pk_value> con el método DELETE. Con estos 5 endpoints cubrimos las operaciones CRUD básicas, y cada uno cuenta con protección contra autodestrucción, Validación de nombres de tabla para prevenir inyección SQL y soporte para claves primarias compuestas con parámetros de query.

  4. Generación de Reportes

    Nuestro sistema es capaz de generar reportes que pueden ser exportados en formato pdf o csv utilizando las librerías jsPDF y Papa Parse. Utilizamos funciones como exportToPDF() que convierte elementos HTML a imágenes usando html2canvas y las integra en documentos PDF multipágina con formato A4, y exportToCSV() que procesa datos médicos complejos (arrays de arrays) y los convierte a objetos estructurados con nombres de columnas descriptivos específicos para cada tipo de reporte

  5. Validación de Datos

    El sistema valida los datos de todos los formularios, en distintas capas. Inicialmente se asegura que ningún formulario pueda tener espacios en blanco con HTML en el front. Así como también tenemos validaciones para las claves de las operaciones CRUD que podemos realizar para prevenir SQL injection. Realizamos validación de JWT con manejo de errores para asegurarnos que los usuarios estén siempre autenticados de manera correcta. Nos aseguramos de agregar mensajes de error claros para guiar al usuario, y de mantener la integridad y seguridad de todos los datos.

  6. Conexión directa con la base de datos

    El proyecto implementa conexión directa entre el backend y la base de datos sin intermediarios, utilizando psycopg2 como driver nativo de PostgreSQL que establece conexión directa desde Flask a PostgreSQL.

Queries SQL y Stored Procedures

Stored Procedures de Autenticación y Usuarios

  • sp_login_staff - Auténtica que el usuario esté autorizado (Administradores y Analistas)

    Utilizando la tabla Usuario valida las credenciales, verifica el rol del usuario, registra su IP, y regresa la información del usuario autenticado.

  • sp_get_user_data - Obtiene los datos completos de un usuario autenticado

    Primeramente valida que el usuario exista, y regresa su email, rol, nombre del rol e información para su token JWT.

Stored procedures de Gestión de Entidades

  • sp_get_entidades - Lista todas las tablas de la base de datos.

    Consulta la tabla Entidad, y regresa el nombre y ID de cada tabla.

  • sp_get_entidad_data_with_columns - Obtiene datos y metadatos de una entidad específica.

    Regresa todos los registros de una tabla especifica, metadatos de las columnas, y puede utilizarse para tablas con nombres dentro de comillas.

  • sp_get_table_columns - Obtiene columnas de una tabla.

    Consulta el esquema information_schema.columns y regresa los nombres de las columnas ordenadas por posición. La utilizamos para validación en campos en las operaciones CRUD.

Stored Procedures de Operaciones CRUD

  • sp_insert_record - Inserción genérica de registros en cualquier tabla.

    Valida que la columna si exista, crea un INSERT y después regresa el ID del registro insertado. Además tiene manejo de errores con rollback.

  • sp_update_by_pk - Actualización de la tabla por clave primaria

    Construye un UPDATE que filtra por clave primaria y solo actualiza los campos especificados. Regresa el número de registros actualizados.

  • sp_update_by_pk_multi - Actualización por clave primaria compuesta

    Maneja claves primarias múltiples, construye un WHERE con múltiples condiciones, y está diseñado para soportar tablas con Primary Keys compuestas.

  • sp_delete_by_pk - Elimina registros por la clave primaria

    Filtra por clave primaria, construye un delete y regresa el número de registros eliminados.

  • sp_delete_by_pk_multi - Elimina registros por clave primaria compuesta

    Maneja múltiples claves primarias, construye un WHERE con múltiples condiciones, y esta diseñada para soportar tablas con Primary Keys compuestas.

Stored Procedures de Dashboards

  • sp_dashboard_kpis - Regresa indicadores clave de rendimiento del sistema

    Regresa información sobre total de usuarios, documentos, predicciones, usuarios activos por periodo y crecimiento semanal.

  • sp_dashboard_monitoreo_pa - Regresa datos de monitoreo de presión arterial

    Regresa información sobre valores sistólicos y diastólicos, promedios, mínimos y máximos, y análisis de tendencias.

  • sp_dashboards_signos_vitales - Consolida los signos vitales

    Regresa información de la frecuencia cardíaca, saturación de oxígeno, análisis de variabilidad y promedios por periodo.

  • sp_dashboard_lab - Regresa información sobre los resultados de laboratorios

    Regresa información sobre análisis clínicos, valores de referencia, tendencias temporales y alertas por valores anómalos.

  • sp_dashboard_estilo_de_vida - Recupera las respuestas de cuestionarios de estilo de vida.

    Regresa información con respuestas de si/no, valores numéricos y escalas, promedios por preguntas y análisis de patrones.

  • sp_dashboard_predicciones - Regresa información de predicciones de IA

    Regresa información sobre la probabilidad de enfermedades, nivel de confianza de las predicciones, tendencias y análisis de riesgo cardiovascular.

Stored Procedures de Gráficos Especializados

  • sp_dashboard_predicciones_por_mes - Regresa información sobre las predicciones agrupadas

    Regresa información sobre el total de predicciones por mes, probabilidad promedio, predicciones positivas/negativas, y tendencias temporales.

  • sp_dashboard_distribucion_enfermedades - Regresa información sobre la distribución de enfermedades detectadas

    Regresa información sobre el conteo por tipo de enfermedad, porcentajes de distribución, análisis demográfico y tendencias epidemiológicas.

  • sp_dashboard_estado_documentos - Regresa información sobre el estado de procesamiento de documentos

    Regresa información sobre documentos pendientes, documentos procesados, documentos con errores, y tiempo de procesamiento.

  • sp_dashboard_distribucion_demografica - Regresa información sobre la distribución demográfica de usuarios

    Regresa información sobre los usuarios en agrupaciones de sexo y edad, análisis de cohortes, tendencias demográficas y segmentación de usuarios.

  • sp_dashboard_crecimiento_acumulado_usuarios - Regresa información sobre el crecimiento acumulado de usuarios

    Regresa información sobre los usuarios nuevos por mes, usuarios acumulados, tasa de crecimiento y análisis de retención.

  • sp_dashboard_top_usuarios_activos - Regresa información sobre top usuarios más activos

    Regresa información sobre el ranking de actividad, métricas de engagement, análisis de comportamiento e identifica power users.

Stored Procedures de Auditoría

  • sp_insert_audit_record - Registra acciones de auditoría

    Regresa el log de todas las operaciones, timestamp automático y trazabilidad completa.

  • sp_get_audit_records - Realiza análisis de registros de auditoría

    Regresa las estadísticas por período, usuario más activo, entidad más accedida y análisis de patrones de uso.

Stored Procedures de Utilidades

  • sp_table_exists - Verifica la existencia de una tabla

    Valida el nombre de la tabla, y nos ayuda a prevenir errores.

  • sp_get_primary_key_column - Obtiene columna de una clave primaria

    Detecta automáticamente una Primary Key, funciona con PK simples y nos ayuda con la validación de integridad.

  • sp_get_primary_key_columns - Obtiene columnas de claves primarias compuestas

    Detecta automáticamente primary keys compuestas, y también nos ayuda con la validación de integridad.

  • sp_get_entity_id_by_name - Obtener ID de la tabla por nombre

    Mapea nombres a IDs, ayuda con la auditoría y con la validación de tablas.

KPIs, Métricas y Gráficas

KPIs principales del sistema

Métricas de Usuarios

  • Total de Usuarios: Número total de usuarios registrados en el sistema
  • Usuarios Activos (30 días): Usuarios que han interactuado con el sistema en el último mes
  • Usuarios Activos (7 días): Usuarios que han interactuado con el sistema en la última semana
  • Usuarios Nuevos (30 días): Nuevos registros en el último mes

Métricas de Contenido

  • Documentos Subidos: Total de documentos médicos procesados
  • Predicciones Generadas: Número de predicciones de IA realizadas
  • Registros Médicos: Total de registros de historial médico
  • Respuestas de Estilo de Vida: Cuestionarios completados por usuarios
  • Signos Vitales Registrados: Mediciones de signos vitales capturadas

Métricas de Actividad

  • Documentos Esta Semana: Documentos subidos en los últimos 7 días
  • Predicciones Esta Semana: Predicciones generadas en los últimos 7 días
  • Registros Médicos Esta Semana: Registros médicos creados en los últimos 7 días
  • Frecuencia Diaria: Número de actualizaciones por día
  • Crecimiento Semanal: Incremento semanal de usuarios y contenido
  • Actividad de Usuarios: Patrones de uso diarios y semanales

Métricas de Calidad

  • Tasa de Procesamiento Exitoso: Porcentaje de documentos procesados correctamente
  • Precisión de Predicciones: Exactitud de las predicciones de IA
  • Tiempo de Respuesta: Velocidad de procesamiento de solicitudes

Métricas de Seguridad

  • Registros de Auditoría: Número de acciones registradas
  • Accesos por Usuario: Frecuencia de acceso por usuario
  • Entidades Más Accedidas: Tablas más consultadas en el sistema

Gráficas Especializadas

Gráfica 1: Predicciones por Mes (de Líneas)

Utiliza el endpoint: /api/dashboard/graficos/predicciones-por-mes, y muestra el total de predicciones, predicciones positivas y negativas para identificar patrones estacionales, picos de actividad y la efectividad del sistema de IA.

Gráfica 2: Distribución de Enfermedades (de Pastel)

Utiliza el endpoint /api/dashboard/graficos/distribucion-enfermedades, y muestra el total de usuarios que tiene diabetes o hipertensión. Esto con la finalidad de ayudar a los doctores a entender a la población que está siendo atendida.

Gráfica 3: Estado de Documentos (de Barras Apiladas)

Utiliza el endpoint /api/dashboard/graficos/estado-documentos, y muestra la cantidad de documentos que ya han sido procesados y los que están pendientes. Esta gráfica nos puede ayudar a identificar si existen problemas con el procesamiento de documentos.

Gráfica 4: Distribución Demográfica (de Barras Horizontales)

Utiliza el endpoint /api/dashboard/graficos/distribucion-demografica, y muestra la composición demográfica de los pacientes, lo cual permite evaluar la representatividad de la población y entender el perfil de los usuarios del sistema.

Gráfica 5: Crecimiento Acumulado de Usuarios (de Área)

Utiliza el endpoint /api/dashboard/graficos/crecimiento-acumulado-usuarios, y distingue entre los usuarios acumulados y los nuevos, además nos permite calcular la velocidad de adopción del sistema.

Gráfica 6: Top Usuarios Activos (Barras Verticales)

Utiliza el endpoint /api/dashboard/graficos/top-usuarios-activos, y muestra los 5 usuarios más activos y comprometidos con el sistema, mide la participación a través de documentos subidos, signos vitales registrados, predicciones realizadas en base a su salud y participación total.


Dashboards Especializados

Dashboard de Monitoreo de Presión Arterial

Utiliza el endpoint /api/dashboard/monitoreo-pa, que utiliza el stored procedure sp_dashboard_monitoreo_pa para tener acceso a los datos completos de monitoreo de presión arterial sin procesamiento adicional. Mientras que utilizamos el endpoint /api/dashboard/monitoreo-pa-stats es una función de análisis estadístico que procesa datos de monitoreo de presión arterial para generar métricas clave como presión sistolica promedio, presión diastolica promedio, presión sistolica minima, etc.

Dashboard de Signos Vitales

Utiliza el endpoint /api/dashboard/signos-vitales, utilizando el stored procedure sp_dashboard_signos_vitales para acceder a los datos completos de signos vitales de los pacientes en el sistema de PredictHealth.

Dashboard de Laboratorios

Utiliza el endpoint /api/dashboard/lab, utilizando el stored procedure sp_dashboard_lab. Su función es acceder a los datos completos de análisis de los laboratorio de los pacientes en el sistema. Después utilizando el endpoint /api/dashboard/lab/stats hacemos el análisis estadístico avanzado que procesa datos de laboratorio para generar métricas clínicas.

Dashboard de Estilo de vida

Utiliza el endpoint /api/dashboard/estilo-vida, y el stored procedure sp_dashboard_estilo_vida nos permite ver los datos completos de cuestionarios de estilo de vida de los pacientes en el sistema. Y con el endpoint /api/dashboard/estilo-vida-stats generamos el análisis estadístico y con el endpoint /api/dashboard/estilo-vida-charts las gráficas visuales de la información obtenida.

Dashboard de Predicciones

Utiliza el endpoint /api/dashboard/predicciones, y el stored procedure sp_dashboard_predicciones nos permite obtener los datos completos de predicciones de modelos de inteligencia artificial.

Dashboard de Medicación

Utiliza el endpoint /api/dashboard/medicacion, y el stored procedure sp_dashboard_medicacion nos permite tener acceso a los datos completos de medicamentos y tratamientos farmacológicos de los pacientes.

Dashboard de Documentos

Utiliza el endpoint /api/dashboard/documentos, y el stored procedure sp_dashboard_documentos nos permite ver los datos completos de documentos médicos digitales almacenados en el sistema.

Dashboard de Auditoría

Utiliza el endpoint /api/dashboard/auditoria, y el stored procedure sp_dashboard_auditoria que nos permite tener acceso al registro de auditoría del sistema para trazabilidad de acciones, detección de anomalías, cumplimiento de regulaciones de salud, y análisis de seguridad del sistema.

Dashboard Completo

Utiliza el endpoint /api/dashboard/completo, y el stored procedure sp_dashboard_completo para proporcionar acceso a una vista consolidada y unificada de todos los dashboards del sistema.

Las métricas de usuarios activos y crecimiento acumulado permiten evaluar la adopción del sistema y identificar tendencias de crecimiento.

Los KPIs de actividad y el ranking de usuarios activos ayudan a entender qué tan comprometidos están los usuarios con el sistema.

Análisis de Salud Pública

Las distribuciones de enfermedades y demográficas proporcionan insights valiosos sobre la salud de la población atendida.

Análisis de Eficiencia

Las métricas de procesamiento de documentos y predicciones permiten evaluar la eficiencia del sistema e identificar áreas de mejora.

Análisis de Calidad

Los indicadores de precisión y tiempo de respuesta ayudan a mantener la calidad del servicio y la satisfacción del usuario. Este conjunto completo de KPIs, métricas y gráficas proporciona una visión integral del sistema PredictHealth, permitiendo a los administradores y analistas tomar decisiones informadas basadas en datos reales y tendencias identificables.

Conclusiones

El desarrollo del módulo del backend permitió consolidar el entendimiento sobre la diferencia entre aplicaciones monolíticas y aplicaciones basadas en microservicios. Se pudo observar con mayor claridad cómo el frontend del cliente web únicamente consume los endpoints o rutas expuestas por el backend, los cuales pueden ser fácilmente sustituidos por los endpoints de los microservicios, siempre y cuando se mantenga la coherencia en los nombres y estructuras de dichos endpoints.

De igual manera, se apreciaron las ventajas de utilizar microservicios en lugar de una aplicación monolítica, ya que conforme se implementaban más rutas, el código del backend monolítico aumentaba considerablemente, generando una mayor complejidad visual y lógica. Esto provocaba que el mantenimiento del proyecto resultara más difícil, especialmente al intentar localizar errores o entender la estructura del sistema. En contraste, con el enfoque de microservicios, el código se percibe más limpio, modular y sencillo de comprender, lo cual facilita la detección de fallos y la escalabilidad del proyecto.

Por otro lado, uno de los principales retos enfrentados fue la automatización de la creación del backend mediante inteligencia artificial. Se observó que las herramientas de IA presentan limitaciones cuando se les asignan tareas demasiado amplias o con múltiples dependencias, lo que llevó a invertir una cantidad considerable de tiempo en desarrollar la primera instancia funcional de cada componente. Una vez completada esta fase, fue posible replicar las demás funcionalidades de forma más eficiente, al comprender la estructura general y los patrones que debían seguirse.

Asimismo, las lecciones aprendidas incluyeron la importancia de redactar prompts más específicos y proporcionar un contexto más detallado a la inteligencia artificial para obtener resultados precisos. También se comprendió la conveniencia de realizar los procesos de manipulación de datos directamente en el backend, dejando al frontend únicamente la tarea de consumir la información.

Contar con una base de datos completa y bien estructurada resultó esencial para mantener un control adecuado sobre la aplicación, especialmente al considerar su uso por parte de usuarios finales, donde será fundamental el monitoreo constante para detectar oportunidades de mejora o corregir errores. Finalmente, se reafirmó la relevancia de utilizar microservicios para evitar la sobrecarga de peticiones desde el frontend, ya que en algunos casos las solicitudes concurrentes provocaban retrasos o fallos en las respuestas cuando el sistema trabajaba de manera monolítica.
