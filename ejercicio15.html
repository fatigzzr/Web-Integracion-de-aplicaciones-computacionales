Ejercicio 15: Prueba ramp-up / ramp-down

Usar LoadTestShape para simular crecimiento y decrecimiento gradual.

Introducción

En este ejercicio se evaluará el desempeño de los microservicios que conforman la aplicación PredictHealth, una plataforma orientada al análisis y predicción de riesgos en salud. El sistema está compuesto por diversos microoservicios que operan de forma coordinada, entre ellos:

  • auth: encargado de la autenticación y emisión de tokens.
  • register: gestión del registro de nuevos usuarios.
  • patient: administración de perfiles y expedientes de pacientes.
  • health: operaciones generales de información médica.
  • diabetes: motor de predicción y análisis de riesgo para diabetes.
  • hypertension: procesamiento de predicciones y evaluaciones de hipertensión.
  • data: manejo de datos clínicos.
  • recommendations: generación de recomendaciones personalizadas para el usuario.

Dado que PredictHealth depende de la interacción simultánea entre múltiples microservicios, resulta fundamental validar que cada uno mantenga un comportamiento estable, eficiente y escalable bajo distintos niveles de carga. Las pruebas de rendimiento permiten identificar cuellos de botella, medir tiempos de respuesta, determinar la tolerancia al tráfico inesperado y garantizar que el sistema pueda sostener un uso real en producción.

Estas pruebas son especialmente importantes en aplicaciones del sector salud, donde los fallos pueden impactar directamente en la disponibilidad y confiabilidad de la información médica. Por ello, la ejecución sistemática de pruebas baseline, smoke, ramp-up, spike, soak y break-point constituye un paso crítico para asegurar que los microservicios de PredictHealth operen correctamente ante escenarios exigentes y condiciones cambiantes de carga.

Objetivos Específicos

A continuación, se presentan los objetivos específicos de cada tipo de prueba de rendimiento que se aplicará a los microservicios del sistema:

4. Prueba Ramp-Up / Ramp-Down (LoadTestShape)

  • Analizar la respuesta del sistema ante incrementos graduales de usuarios concurrentes.
  • Identificar el punto en que comienzan a aparecer degradaciones en el rendimiento.
  • Evaluar la capacidad del sistema para recuperarse cuando la carga disminuye progresivamente.

Metodología

  1. Arquitectura del entorno de prueba: se configuró una arquitectura basada en microservicios independientes, cada uno ejecutándose en un puerto diferente en localhost. El generador de carga (Locust) interactuó directamente con los servicios mediante solicitudes HTTP.
  2. Herramientas utilizadas:
    • Locust: herramienta principal para la generación de carga y ejecución de pruebas de rendimiento.
    • Lenguaje de scripting: Python.
    • Sistema operativo: macOS, Linux o Windows.
    • Formato de resultados: archivos CSV generados y reportes HTML para visualización ejecutiva.
  3. Descripción de los escenarios de carga:
    • Prueba Ramp-Up
      • Usuarios definidos por parámetro: incrementos progresivos
      • Tasa de spawn: ajustada manualmente durante ejecución
      • Descripción: Aunque el LoadTestShape estaba deshabilitado, se simuló un aumento gradual de usuarios modificando los parámetros --users y --spawn-rate, permitiendo observar degradación progresiva del rendimiento.
    • Prueba Ramp-Down
      • Usuarios definidos por parámetro: reducciones progresivas
      • Descripción: Similar a la prueba anterior, pero enfocada en evaluar la recuperación del sistema cuando la carga disminuye de forma gradual.

class PredictHealthUser(HttpUser):
    """Base user class with authentication and common tasks"""
    
    abstract = True  # Don't run this directly
    host = "http://localhost"  # Base host (services use absolute URLs with specific ports)
    access_token: Optional[str] = None
    user_id: Optional[int] = None
    user_email: Optional[str] = None
    
    def on_start(self):
        """Called when a simulated user starts - perform login"""
        self.login()
    
    def login(self):
        """Authenticate and get access token"""
        # Pick random test user
        user_creds = random.choice(TEST_USERS)
        
        response = self.client.post(
            f"{get_service_url('auth')}/auth/login",
            json={
                "username": user_creds["email"],
                "password": user_creds["password"]
            },
            name="/auth/login"
        )
        
        if response.status_code == 200:
            data = response.json()
            self.access_token = data.get("access_token")
            self.user_email = user_creds["email"]
            
            # Get user ID from /auth/me
            if self.access_token:
                me_response = self.client.get(
                    f"{get_service_url('auth')}/auth/me",
                    headers={"Authorization": f"Bearer {self.access_token}"},
                    name="/auth/me"
                )
                if me_response.status_code == 200:
                    me_data = me_response.json()
                    self.user_id = int(me_data.get("sub", 1))
        else:
            # If login fails, use fallback
            self.user_id = 1
            self.access_token = None
    
    def get_auth_headers(self):
        """Return Authorization header dict"""
        if self.access_token:
            return {"Authorization": f"Bearer {self.access_token}"}
        return {}

                    reader = csv.DictReader(f)

    Tanto en Ramp-Up como en Ramp-Down, el servicio más sensible continúa siendo auth, particularmente el endpoint POST /auth/login.   2. ¿Cuál es el punto de ruptura de cada servicio?

    A partir de los patrones observados: (más débil a más resistente)

    • Auth/login: se acerca a ruptura incluso con carga gradual. Sus picos >11 s en Ramp-Down sugieren que su breakpoint está muy próximo en escenarios reales de concurrencia moderada.

    • GET endpoints: permanecen estables aun cuando aumenta o disminuye la carga, su punto de ruptura es más alto y no se alcanza en estas pruebas.
  3. ¿El CMS se comporta igual que los microservicios bajo carga de escritura?

    Según los resultados de Ramp-Up y Ramp-Down, las rutas de escritura presentan los mayores tiempos y degradaciones. Si el CMS realiza operaciones comparables (crear contenido, actualizar información, guardar formularios), entonces su comportamiento sería equivalente al de los endpoints POST, no al de los GET.

Conclusiones y recomendaciones

Los resultados de las pruebas Ramp-Up y Ramp-Down muestran que el sistema responde adecuadamente al incremento y decremento progresivo de carga, pero revelan claramente cuáles microservicios poseen menor tolerancia.

La comparación entre Ramp-Up y Ramp-Down permite observar un patrón importante: incluso cuando la carga disminuye, algunos endpoints(especialmente autenticación) no recuperan inmediatamente su rendimiento, sino que mantienen o incluso incrementan sus tiempos máximos de respuesta. Este comportamiento sugiere degradación acumulada, consistente con posibles problemas de manejo de conexiones, threads ocupados o tareas pendientes que no se liberan con rapidez.

Como recomendaciones, se vuelve prioritario optimizar el servicio de autenticación, incluyendo el uso de caching, reducción del costo de validación y un mejor manejo del pool de conexiones. Para los servicios de escritura, se sugiere optimizar la interacción con la base de datos, reducir operaciones costosas y analizar oportunidades
